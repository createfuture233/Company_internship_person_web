## 1. 核心观点

AI 不应被当作一次性代写 PRD 的工具，而应作为协作伙伴：帮助补全思考、搭建结构、优化表达、发现遗漏与提出行业常见做法。需求负责人仍需掌握目标、约束、优先级和最终决策权。

不推荐的方式是只给出一句宽泛指令，例如“为某类产品写一份 PRD”。这样得到的往往是通用模板，缺少真实场景、业务规则、异常处理、系统依赖和可验收标准，后续也难以基于反馈迭代。

推荐的原则是：先准备事实，再确定结构；按章节生成并逐轮补全；以多角色审查初稿；最后处理反馈、统一口径和定稿。

## 2. PRD-AI 五步工作流

```text
准备信息 → 设计结构 → 分章详述 → 多角度审核 → 整合反馈并发布
```

| 阶段 | 目的 | 人负责的工作 | AI 可协助的工作 | 产出 |
| --- | --- | --- | --- | --- |
| Prepare（准备） | 形成问题定义 | 明确目标、用户、范围和约束 | 追问遗漏信息、生成访谈问题 | 需求事实清单 |
| Restructure（结构） | 确定文档骨架 | 选择适合团队和项目的章节 | 提供章节建议与每章写作要点 | PRD 目录 |
| Detail（详述） | 形成可实施说明 | 提供场景、规则、优先级和决策 | 分章节扩写、给出替代方案 | PRD 初稿 |
| Audit（审核） | 找到缺口与冲突 | 组织评审、判断风险 | 用开发、设计、测试等视角质疑文档 | 问题清单与修订项 |
| Integrate（整合） | 达成可执行共识 | 做取舍、解决冲突、确认版本 | 汇总反馈、统一语言与格式 | 定稿 PRD |

### 2.1 第一步：准备信息

在使用 AI 前，先写清楚以下问题：

- 要解决的用户或业务问题是什么？为什么现在要做？
- 目标用户是谁，他们处于什么使用情境？
- 如何判断成功？包括业务指标、体验指标或交付标准。
- 有哪些不可突破的限制：时间、预算、人力、技术、合规、数据或既有系统？
- 当前已有的功能、流程、数据和竞品信息是什么？

信息可来自短访谈、已有业务文档、数据分析、竞品体验和现有系统调研。

可用提示词结构：

```text
我需要为【功能/产品】编写 PRD。
目标是【目标】；目标用户是【用户与场景】；要解决的问题是【问题】。
现有限制包括【时间、资源、技术、合规等】。
请不要直接生成 PRD。请先列出为完成 PRD 仍需确认的 10—15 个关键问题，
并按“业务、用户、流程、数据、技术/合规、验收”分类。
```

### 2.2 第二步：设计结构

PRD 没有唯一模板，应随项目复杂度、团队协作习惯、开发模式和公司规范调整。常用章节如下：

1. 文档信息：版本、作者、变更记录、相关链接。
2. 背景与目标：问题、机会、业务价值、目标与非目标。
3. 用户与场景：用户画像、使用情境、用户旅程、核心任务。
4. 功能规格：功能模块、规则、状态、输入输出、优先级。
5. 交互与界面：页面信息架构、流程、关键状态、视觉约束。
6. 技术与集成：已有系统、接口、数据流、依赖与限制。
7. 非功能需求：性能、安全、兼容性、可访问性、可用性、国际化等。
8. 计划与风险：里程碑、范围、依赖、风险和回滚策略。
9. 验收与度量：验收条件、测试方式、成功指标、埋点或数据计划。
10. 附录：术语、原型、流程图、参考资料。

可用提示词结构：

```text
请为【功能/产品】设计一份 PRD 目录。
项目复杂度为【简单/中等/复杂】，属于【新建/优化/改造】；
团队采用【敏捷/瀑布/其他】方式，参与角色有【角色】，已有规范是【规范】。
请列出一级、二级目录，并说明每一章要回答的问题和应包含的关键信息。
不要加入与该项目无关的通用章节。
```

### 2.3 第三步：按章节详述

不要让 AI 一次性写完整文档。先处理最关键的用户旅程和核心功能，再逐章扩展。每一轮都要给 AI 足够上下文，并要求输出可比较的选项或清晰的表格。

功能规格章节至少应覆盖：

- 功能目标与优先级；
- 用户角色、前置条件与触发方式；
- 主流程、分支流程和结束状态；
- 字段、输入输出、状态变化和业务规则；
- 异常、边界和失败后的反馈或恢复；
- 与其他模块、接口或数据的交互；
- 验收标准与可观测指标。

可用提示词结构：

```text
我正在编写【功能】的“功能规格”章节。
功能目标是【目标】，主要场景为【场景列表】，相关约束是【约束】。
请按功能模块输出：用户动作、系统响应、输入输出、业务规则、异常处理、
外部依赖、优先级和验收条件。信息不足处请标为“待确认”，不要自行假设。
```

### 2.4 第四步：审核

初稿不是终稿。审核应从五个维度进行：

| 维度 | 审核问题 |
| --- | --- |
| 完整性 | 目标、角色、流程、规则、异常、依赖、验收是否齐全？ |
| 一致性 | 各章节的术语、规则、数据和优先级是否矛盾？ |
| 清晰度 | 是否存在不同角色会作出不同理解的描述？ |
| 可测试性 | 是否能明确判断“已完成”及如何测试？ |
| 可行性 | 技术、资源、时间和合规限制下能否实现？ |

可让 AI 分别扮演开发、设计、测试、运维、安全或业务负责人审查同一片段；但最终判断必须由了解项目事实的人完成。

```text
请以【角色】的视角审查以下 PRD 内容：
【粘贴内容】
请列出：信息不足之处、冲突或歧义、遗漏风险、实现阻碍，以及可直接写入 PRD 的修订建议。
每项请说明影响范围和优先级；不确定的信息请以问题形式提出。
```

### 2.5 第五步：整合与定稿

收集开发、设计、测试、业务等角色反馈后，按“目标影响、用户价值、实现成本、风险、时间”处理冲突。定稿前应：

- 明确范围内、范围外和后续版本内容；
- 为争议项记录决策结论与责任人；
- 统一术语、编号、格式和版本号；
- 输出摘要、流程图、关键页面或接口附件；
- 确认发布后的变更机制与评审对象。

## 3. 常用场景提示词库

以下是原文覆盖的 15 类场景。使用时补全方括号，并让 AI 先询问缺失信息；不要直接照搬输出。

| 分类 | 场景 | PRD 必须关注的内容 |
| --- | --- | --- |
| 基础功能 | 注册与登录 | 登录方式、字段校验、账号安全、失败与锁定、认证规则 |
| 基础功能 | 数据列表与筛选 | 默认排序、分页、筛选条件、单条/批量操作、空态和大数据量性能 |
| 基础功能 | 表单提交 | 字段类型/必填/校验、字段联动、分步逻辑、保存草稿、提交后的处理 |
| 系统功能 | 权限管理 | 角色模型、页面/操作/数据权限、授权流程、审计日志和拦截机制 |
| 系统功能 | 数据导入导出 | 文件格式、字段映射、校验、异步任务、错误报告、权限与安全 |
| 系统功能 | 消息通知 | 通知类型、触发条件、渠道、频控、偏好设置、已读状态和历史检索 |
| 交互设计 | 移动端导航 | 导航层级、页面跳转、返回策略、快捷入口、不同屏幕适配 |
| 交互设计 | 数据仪表盘 | 指标口径、布局、图表、筛选钻取、交互、导出和分享 |
| 交互设计 | 新手引导 | 步骤、引导方式、进度、跳过与重进、完成标准、数据埋点 |
| 业务流程 | 电商购买 | 商品到订单的全流程、购物车、地址、支付、订单跟踪与异常订单 |
| 业务流程 | 企业审批 | 触发条件、角色、分支规则、通知催办、记录追溯、加急与例外处理 |
| 业务流程 | 注册转化优化 | 漏斗数据、流失点、表单摩擦、激励、实验方案、度量与分析 |
| 特殊场景 | API 文档 | 鉴权、端点、参数、响应、错误码、限流、版本、示例与调用约束 |
| 特殊场景 | 数据迁移 | 迁移策略、映射、清洗转换、校验、监控、回滚、风险和连续性 |
| 特殊场景 | 多语言/国际化 | 语言、日期/货币等地区规则、翻译流程、素材本地化、测试和扩展性 |

### 通用场景提示词骨架

```text
请为【系统/功能】编写需求规格。
使用者是【角色】，他们要完成的任务是【任务】；业务目标和成功指标为【指标】。
已知条件：数据/规模【数据】，技术或合规限制【限制】，关联系统【系统】。
请输出：1) 用户流程；2) 功能与规则；3) 页面/交互状态；4) 异常与边界；
5) 数据、权限和集成；6) 非功能要求；7) 验收标准；8) 待确认问题。
```

### 如何选择与调整模板

1. 先判断功能属于基础功能、系统能力、交互设计、业务流程或特殊场景。
2. 选择最接近的模板，再按项目特点增加或删除约束。
3. 复杂功能可组合多个模板，例如“注册转化优化”同时使用表单、登录和漏斗模板。
4. 补入公司现有系统、品牌规则、数据口径、行业合规要求和团队文档格式。
5. 按团队成熟度调整颗粒度：经验较少时写得更具体，经验成熟时保留必要的决策与边界即可。

## 4. AI 辅助 PRD 的七项风险检查

| 风险 | 典型表现 | 修正方式 |
| --- | --- | --- |
| 内容泛化 | 像任何产品都能套用的模板 | 给出用户情境、差异化价值、竞品和真实业务事实 |
| 功能堆砌 | 有功能清单但无用户问题和闭环 | 从用户目标与旅程组织功能，标注必要/重要/增强优先级 |
| 忽略异常 | 只写正常流程 | 每个主功能至少讨论网络、无效输入、过载、第三方故障、存储不足等失败情形 |
| 业务与技术混淆 | 将具体技术选型写成用户需求 | 先说明“要达到什么效果”，再将“如何实现”放到方案或技术章节 |
| 不可验收 | 使用“快、友好、易用”等模糊描述 | 为每项需求补上成功指标、测试方法、边界期望与性能阈值，尽量使用 SMART 原则 |
| 忽略体验与非功能项 | 只关注功能按钮 | 写清性能、安全、可访问性、兼容性、可用性、国际化，并关联用户画像 |
| 缺乏集成上下文 | 把新功能当孤岛 | 提供当前技术栈、数据库、第三方服务、数据流和接口依赖，说明交互方向 |

### 4.1 异常与边界审查提示词

```text
请为【功能】补充异常与边界处理。至少覆盖：网络中断、非法或恶意输入、
高并发/负载、第三方服务不可用、存储或配额不足、重复提交和权限不足。
对每个场景说明：触发条件、用户提示、系统处理、数据一致性、重试/恢复方式和日志要求。
```

### 4.2 验收标准提示词

```text
请为以下功能逐条生成可验证的验收标准：【功能列表】。
每条必须含成功条件、验证方法、边界输入下的预期结果和必要的性能指标。
避免使用没有度量定义的主观词；如信息不足，请列出待确认的阈值。
```

### 4.3 集成审查提示词

```text
当前环境：前端【技术】；后端【技术】；数据库【类型】；外部服务【列表】。
新功能将与【系统/服务】交互。请识别集成点、数据流、调用时机、权限、
失败降级、数据同步、接口依赖和需要澄清的责任边界。
```

## 5. 学习与能力提升路径

| 阶段 | 建议周期 | 重点 | 练习成果 |
| --- | --- | --- | --- |
| 新手 | 0—2 周 | 理解基本 PRD 结构和提示词；学习接收反馈 | 为简单表单、列表页各写一份 PRD |
| 进阶 | 2—4 周 | 角色扮演审查、多轮对话、发现常见风险 | 为用户管理或多步骤流程写完整文档 |
| 专家 | 4—8 周 | 建个人模板库、处理不确定需求、优化协作流程 | 完成创新功能 PRD，并设计团队协作机制 |

持续改进建议：

- 建立个人提示词库，记录每个模板的适用条件、成功案例和失败原因。
- 每完成 3—5 份文档做一次复盘，更新常见问题和检查清单。
- 轮流从产品、开发、设计、测试、安全、运营等角色审查需求。
- 收集实际使用者反馈，观察需求是否减少返工，再反向修订模板。

## 6. 对 AI 与产品角色的理解

未来的工具会更擅长理解多种输入形式、利用上下文、输出图表与流程图、辅助多人协作、发现需求冲突和维护随项目变化的文档。这不意味着产品人员不再重要：基础撰写和格式整理会减少，而问题定义、战略判断、跨团队协调、AI 输出审校和复杂系统拆解会更加重要。

建议建立“人 + AI”的混合流程：人提供事实、目标、价值判断和最终责任；AI 负责发散、结构化、表达、检查和初步汇总。持续培养提示设计、系统思考、批判性评估、沟通协调与创新整合能力。

## 7. 发布前总检查清单

- [ ] 已说明问题、目标用户、成功指标与非目标。
- [ ] 已写出范围、优先级、限制条件与关联方。
- [ ] 主流程、分支、异常、空态和失败恢复均有描述。
- [ ] 业务需求与实现方案已经分开。
- [ ] 每项关键需求都有可测试的验收标准。
- [ ] 已覆盖性能、安全、可访问性、兼容性和隐私等非功能要求。
- [ ] 已说明现有系统、接口、数据流、依赖和责任边界。
- [ ] 已由开发、设计、测试、业务等相关角色审查。
- [ ] 冲突、待确认项和版本决策均被记录。
- [ ] 文档格式、术语、链接、原型和附件一致且可访问。

## 8. 结语

优秀 PRD 的目标始终是对齐团队、明确问题与成功标准、降低误解和返工。AI 改变的是信息整理和迭代的效率，不替代对用户、业务和决策的理解。把 AI 纳入分阶段、可审查、可追溯的工作流，才能真正提高需求文档质量。
